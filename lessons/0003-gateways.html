Lesson 3: Gateways

In the previous lesson, we defined a flow that starts from a channel named "helloChannel". But how do we actually put a message onto that channel from our regular Java code (like a REST Controller or a Service)?

While you could inject a MessageChannel bean and call channel.send(new GenericMessage<>("hello")), Spring Integration provides a much cleaner, more decoupled pattern: The Messaging Gateway.

The @MessagingGateway Interface

A Gateway hides the messaging API from your application code. You define a plain Java interface, and Spring generates a proxy implementation for it at runtime.

import org.springframework.integration.annotation.MessagingGateway;

@MessagingGateway(defaultRequestChannel = "helloChannel")
public interface MyGateway {
    
    // When you call this method, Spring will take the String "data", 
    // wrap it in a Message, and send it to "helloChannel"
    void sendToFlow(String data);
}

Why is this awesome?

  • Decoupling: The rest of your application just calls a standard Java method on an interface. It has zero dependency on Spring Integration classes like Message or MessageChannel.
  • Ease of Testing: You can easily mock this interface in your unit tests.

Putting it together

Now, in your business logic, you just inject the interface and call it:

import org.springframework.stereotype.Service;

@Service
public class MyBusinessService {

    private final MyGateway myGateway;

    public MyBusinessService(MyGateway myGateway) {
        this.myGateway = myGateway;
    }

    public void processData() {
        // This looks like a normal method call, but it's actually 
        // publishing a message to the Spring Integration flow!
        myGateway.sendToFlow("hello spring integration");
    }
}
