Lesson 6: Message Headers

We've successfully routed and processed the Payload (the Order object). But remember Lesson 1? A Message consists of a Payload and Headers.

Headers are metadata (key-value pairs) attached to the message. Why are they useful?

  • Routing: You can route based on a header (e.g., "Priority": "HIGH") instead of having to inspect the payload.
  • Tracking: Adding a correlation ID or timestamp as the message moves through the system.
  • Clean Domain Models: You might need data for the integration flow (like a specific target database schema) that doesn't belong inside your pure Order POJO. Headers keep your POJOs clean.

Adding Headers (Header Enricher)

You add or modify headers using a Header Enricher. In the Java DSL, you use the .enrichHeaders() method.

@Bean
public IntegrationFlow orderFlow(OrderService service) {
    return IntegrationFlow.from("newOrderChannel")
        // Add a static header
        .enrichHeaders(h -> h.header("sourceSystem", "WEB_API"))
        
        // Add a dynamic header based on the payload
        .enrichHeaders(h -> h.headerFunction("isLargeOrder", 
            message -> {
                Order order = (Order) message.getPayload();
                return order.getQuantity() > 100;
            }))
            
        .handle(service, "process")
        .get();
}

Reading Headers in your Service

How does your Service Activator access these headers if it only takes the Order as an argument? You use the standard Spring Messaging @Header annotation!

@Service
public class OrderService {
    
    // Spring extracts both the payload AND the specific header for you!
    public void process(Order order, @Header("isLargeOrder") boolean large) {
        if (large) {
            System.out.println("Processing LARGE order...");
        } else {
            System.out.println("Processing normal order...");
        }
    }
}
