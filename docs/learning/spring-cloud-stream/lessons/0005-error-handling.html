Lesson 5: Error Handling and Resilience

In the real world, things fail. A database might be down, an API might timeout, or a message payload might be malformed. What happens when your Consumer throws a RuntimeException?

Default Behavior: Retries

By default, if your function throws an exception, Spring Cloud Stream will retry processing the message 3 times (with a backoff of 1 second between attempts).

If all 3 retries fail, the message is dropped by default. This is often dangerous in production because you silently lose data!

Configuring a Dead Letter Queue (DLQ)

To prevent data loss, we configure a Dead Letter Queue. A DLQ is a special topic/queue where messages are sent if they cannot be processed successfully after all retries are exhausted.

You enable DLQs via the specific binder configuration. For example, if you are using RabbitMQ or Kafka, you configure it under the binding's binder-specific properties.

Example: Enabling DLQ for a Consumer

spring:
  cloud:
    stream:
      bindings:
        processOrder-in-0:
          destination: incoming-orders
          group: fulfillment-group
      
      # Binder-specific configuration to enable DLQ
      kafka:
        bindings:
          processOrder-in-0:
            consumer:
              enableDlq: true
              dlqName: incoming-orders-dlq

Note: If you were using RabbitMQ, the property is autoBindDlq: true instead of enableDlq.

Customizing Retries

You can also customize the retry logic (e.g., increase max attempts, or change the backoff) using core Spring Cloud Stream properties:

spring:
  cloud:
    stream:
      bindings:
        processOrder-in-0:
          consumer:
            max-attempts: 5
            back-off-initial-interval: 2000 # 2 seconds
