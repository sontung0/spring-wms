Binder Detection

How Spring Cloud Stream figures out which binder to actually use.

1. Classpath Detection

You learned in Lesson 0009 that a binder registers itself via a META-INF/spring.binders file. But how does Spring Cloud Stream decide which registered binder to actually activate for your application?

By default, Spring Cloud Stream relies on Spring Boot's auto-configuration to drive this. The rule is simple:

The rule: If a single Binder implementation is found on the classpath, Spring Cloud Stream automatically uses it. No configuration required.

For example, if you want your application to bind only to RabbitMQ, you simply add the RabbitMQ binder as a dependency:

<dependency>
  <groupId>org.springframework.cloud</groupId>
  <artifactId>spring-cloud-stream-binder-rabbit</artifactId>
</dependency>

Because this is the only binder on the classpath, Spring Cloud Stream detects it automatically and wires it up — you never have to tell it "use RabbitMQ" explicitly.

Connect the dots: This is exactly why swapping from Kafka to RabbitMQ (as mentioned in Lesson 0008) can be "just a dependency change." Remove the Kafka binder JAR, add the RabbitMQ binder JAR, and classpath detection does the rest — as long as only one binder is present.

What happens when more than one binder is on the classpath at the same time? That's not something automatic detection can resolve on its own — and it's exactly the subject of the next lesson.
