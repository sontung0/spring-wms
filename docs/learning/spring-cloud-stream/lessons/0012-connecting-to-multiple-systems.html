Connecting to Multiple Systems

Two brokers, same type, different hosts — all in one application.

1. Beyond "Which Type" — "Which Instance"

Lesson 0011 solved routing bindings to different types of binder (Kafka vs. RabbitMQ). But what if you need to talk to two separate instances of the same broker type — say, two different RabbitMQ clusters?

By default, binders share the application's Spring Boot auto-configuration, so only one instance of each binder type on the classpath gets created. To connect to more than one broker of the same type, you must define multiple named binder configurations, each with its own environment settings.

Important side effect: Turning on explicit binder configuration disables the default binder configuration process altogether. Once you do this, every binder your application uses must be explicitly included in the configuration — there's no more "free" auto-detected binder.

2. Example: Two RabbitMQ Instances

spring:
  cloud:
    stream:
      bindings:
        input:
          destination: thing1
          binder: rabbit1
        output:
          destination: thing2
          binder: rabbit2
      binders:
        rabbit1:
          type: rabbit
          environment:
            spring:
              rabbitmq:
                host: <host1>
        rabbit2:
          type: rabbit
          environment:
            spring:
              rabbitmq:
                host: <host2>

Here, rabbit1 and rabbit2 are both configuration names — not binder types. Each has its own type: rabbit plus its own isolated environment block (in this case, a different host). The input binding uses rabbit1, while output uses rabbit2 — same binder type, different broker instances.

3. The environment Block Is Just Spring Boot Properties

The environment property under each named binder configuration can hold any Spring Boot property scoped to that binder — not just broker connection details. For example, you can override auto-configured beans by pointing at a custom configuration class:

environment:
    spring:
        main:
           sources: com.acme.config.MyCustomBinderConfiguration

Or activate a dedicated Spring profile just for that binder's environment:

environment:
    spring:
        profiles:
           active: myBinderProfile

4. Opting Out of the Default Binder Process

Some frameworks want to define binder configurations that Spring Cloud Stream can reference by name, without those configurations affecting (or being affected by) the default binder auto-configuration. This is done with the defaultCandidate flag:

spring.cloud.stream.binders.<configurationName>.defaultCandidate=false

This marks a binder configuration as existing independently of the default binder configuration process.
