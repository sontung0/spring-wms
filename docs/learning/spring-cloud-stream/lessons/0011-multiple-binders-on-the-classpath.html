Multiple Binders on the Classpath

What happens when auto-detection can't pick a winner.

1. The Problem

Lesson 0010 established the rule: with exactly one binder on the classpath, Spring Cloud Stream auto-selects it. But what if your application needs to talk to both Kafka and RabbitMQ at once? Now there are two META-INF/spring.binders registrations, and the framework can no longer guess which one to use for which binding.

The rule: When multiple binders are present on the classpath, the application must indicate which binder to use for each destination binding. Auto-detection alone is not enough.

2. Global Selection: defaultBinder

If you want one binder to be the default for anything that doesn't specify otherwise, set:

spring.cloud.stream.defaultBinder=rabbit

This applies globally, across all bindings that don't override it individually.

3. Per-Binding Selection

More often, you want fine-grained control — for example, a processor that reads from Kafka and writes to RabbitMQ. You configure the binder per-binding, using the binding name plus .binder:

spring.cloud.stream.bindings.input.binder=kafka
spring.cloud.stream.bindings.output.binder=rabbit

Here, a processor with bindings named input (read) and output (write) is explicitly wired to two different messaging systems in a single application.

Connect the dots: Recall from Lesson 0009 that each binder registers itself under a name (the key in spring.binders, e.g. kafka or rabbit). That same name is what you reference in defaultBinder and in the per-binding .binder property.
