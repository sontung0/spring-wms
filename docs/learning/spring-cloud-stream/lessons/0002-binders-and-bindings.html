Lesson 2: Binders and Bindings

In Lesson 1, we wrote a simple Java Function<String, String> uppercase(). But how does Spring know which Kafka topic or RabbitMQ exchange to listen to, and where to send the result?

This is where Binders and Bindings come in.

The Binder

A Binder is the component responsible for providing integration with the physical messaging system. It's usually provided as a dependency in your pom.xml or build.gradle.

  • For Kafka: spring-cloud-stream-binder-kafka
  • For RabbitMQ: spring-cloud-stream-binder-rabbit

The Binder translates the framework's generic message model into the specific protocol required by the broker.

Bindings and Naming Conventions

A Binding is the bridge between your Java code (the Function) and the physical destination (the Topic/Queue) managed by the Binder.

Spring Cloud Stream uses a strict naming convention to automatically create these bindings based on your bean names.

The pattern is: <functionName>-<in-or-out>-<index>

For our uppercase function:

  • Input binding: uppercase-in-0 (Listens for incoming messages)
  • Output binding: uppercase-out-0 (Sends out the result)

Note: The 0 index is used because a function can technically have multiple inputs/outputs, but for most standard use cases, it's always 0.

Configuration (application.yml)

You map these logical bindings to physical broker destinations using configuration properties:

spring:
  cloud:
    stream:
      bindings:
        uppercase-in-0:
          destination: raw-text-topic
          group: text-processor-group
        uppercase-out-0:
          destination: uppercase-text-topic

Important Properties

  • destination: The actual name of the topic or exchange on the broker.
  • group: The Consumer Group. Always set a group for consumers. If you don't, Spring assigns an anonymous, volatile group, meaning you'll lose messages if your app restarts, and you can't load-balance across multiple instances of your app.
