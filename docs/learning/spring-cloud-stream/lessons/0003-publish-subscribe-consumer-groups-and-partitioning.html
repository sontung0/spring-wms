Publish-Subscribe, Consumer Groups, and Partitioning

How Spring Cloud Stream routes messages to the right application instances.

1. Persistent Publish-Subscribe

Communication between applications in Spring Cloud Stream follows a publish-subscribe model. Data is broadcast through shared topics.

If an application publishes a message to a destination (e.g., raw-sensor-data), multiple independent applications can subscribe to that destination. Each application will receive its own copy of the message. This reduces coupling and allows you to add new applications to the topology without disrupting existing flows.

2. Consumer Groups

What happens when you need to scale up a single application? If you run 3 instances of your "average-calculating" application, you don't want all 3 instances to process the exact same message. You want them to share the load.

Spring Cloud Stream solves this with Consumer Groups (inspired by Kafka). When multiple instances of an application share the same group name, they are placed in a competing consumer relationship. Only one instance in the group will receive a given message.

You configure this via properties:

spring.cloud.stream.bindings.myFunction-in-0.group=average-calculator-group
Rule of Thumb: Every independent microservice should have its own unique consumer group name. If you don't specify a group, Spring Cloud Stream creates an anonymous, single-member group (which means it will receive every message, but won't share the load with other instances).

3. Partitioning

Sometimes, just sharing the load isn't enough. In stateful processing, you need to ensure that all related data is processed by the same consumer instance. For example, all temperature readings from "Sensor A" must go to Instance 1, so it can calculate an accurate rolling average.

This is called Partitioning. Spring Cloud Stream provides a uniform abstraction for partitioning, even if the underlying broker (like RabbitMQ) doesn't support it natively.

Configuring the Producer

The producer must be configured to extract a "partition key" from the message (e.g., the sensor ID) to determine which partition the message belongs to:

spring.cloud.stream.bindings.myFunction-out-0.producer.partitionKeyExpression=headers.sensorId
spring.cloud.stream.bindings.myFunction-out-0.producer.partitionCount=5

Configuring the Consumer

The consumer must be configured to know it is receiving partitioned data. (Note: If you are using Kafka with auto-rebalance enabled, Kafka handles the consumer side automatically!)

spring.cloud.stream.bindings.myFunction-in-0.consumer.partitioned=true
spring.cloud.stream.instanceCount=5
spring.cloud.stream.instanceIndex=0 # This changes per instance (0 to 4)
