Lesson 6: Testing Spring Cloud Stream

In a real-world project, you need automated tests. But spinning up a real Kafka or RabbitMQ container for every unit test is slow and brittle.

Spring Cloud Stream provides a dedicated testing module to solve this: TestBinder.

The TestBinder

The spring-cloud-stream-test-binder dependency gives you a simulated, in-memory broker. It allows you to inject messages into an input binding and inspect messages coming out of an output binding, all within a standard JUnit test.

1. Add the Dependency

First, ensure you have the test binder in your pom.xml (in the <test> scope):

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-stream-test-binder</artifactId>
    <scope>test</scope>
</dependency>

2. Using InputDestination and OutputDestination

In your test class, you autowire two special beans provided by the test binder: InputDestination (to simulate incoming messages) and OutputDestination (to catch outgoing messages).

Let's test our Function<String, String> uppercase from Lesson 1:

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.cloud.stream.binder.test.InputDestination;
import org.springframework.cloud.stream.binder.test.OutputDestination;
import org.springframework.messaging.Message;
import org.springframework.messaging.support.MessageBuilder;
import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest
public class TextProcessorTests {

    @Autowired
    private InputDestination input;

    @Autowired
    private OutputDestination output;

    @Test
    public void testUppercaseFunction() {
        // 1. Create a test message
        Message<byte[]> inputMessage = MessageBuilder.withPayload("hello".getBytes()).build();

        // 2. Send it to the input binding
        input.send(inputMessage, "uppercase-in-0");

        // 3. Receive the result from the output binding (timeout after 1 second)
        Message<byte[]> resultMessage = output.receive(1000, "uppercase-out-0");

        // 4. Assert the result is what we expect
        assertThat(resultMessage).isNotNull();
        assertThat(new String(resultMessage.getPayload())).isEqualTo("HELLO");
    }
}

Why this is powerful

Notice what this test accomplishes:

  • It verifies your business logic (hello became HELLO).
  • It verifies that Spring successfully converted your byte array into a String and back.
  • It runs extremely fast, entirely in memory, without Docker or network dependencies.

Knowledge Check

You have written a Consumer<Order> that saves orders to a database. It does not output any messages back to the broker.

If you were writing a test for this using the Test Binder, which of these tools would you use?

  1. Only OutputDestination
  2. Only InputDestination
  3. Both InputDestination and OutputDestination
Correct! Since a Consumer doesn't produce an output binding, you only use InputDestination to send the message in, and then you would assert against your database repository to verify the order was saved.
Not quite. Remember the definition of a Consumer from Lesson 1: it's a Sink. It takes data in, but does not send data out.
