The Functional Programming Model

Writing message handlers using pure Java functions.

The Shift to Functions

In older versions of Spring Cloud Stream, you had to use framework-specific annotations like @StreamListener. Today, the recommended approach is to use the built-in support for Spring Cloud Function. You simply express your message handlers as beans of type java.util.function.[Supplier/Function/Consumer].

The Three Core Interfaces

1. Function (Processor)

A Function<I, O> takes an input and produces an output. It acts as a processor in the middle of a stream.

@Bean
public Function<String, String> toUpperCase() {
    return s -> s.toUpperCase();
}

2. Consumer (Sink)

A Consumer<I> takes an input but returns nothing. It is the end of the line (a sink), like saving to a database or printing to the console.

@Bean
public Consumer<String> sink() {
    return System.out::println;
}

3. Supplier (Source)

A Supplier<O> takes no input but produces an output. It is the origin of data.

@Bean
public Supplier<String> stringSupplier() {
    return () -> "Hello from Supplier";
}
How is a Supplier triggered? Because a Supplier has no input to trigger it, the framework provides a default polling mechanism. By default, it will invoke your Supplier's get() method every 1 second (1000ms) and send the result to the broker.

Functional Binding Names

How does the framework know which Kafka topic to connect your function to? It uses a strict naming convention for bindings:

  • Input bindings: <functionName>-in-<index>
  • Output bindings: <functionName>-out-<index>

For the toUpperCase function above, the framework automatically creates two bindings: toUpperCase-in-0 and toUpperCase-out-0. (The index is 0 because it's a simple function with one input and one output).

You then configure the actual destinations in your application.yml:

spring.cloud.stream.bindings.toUpperCase-in-0.destination=raw-data-topic
spring.cloud.stream.bindings.toUpperCase-out-0.destination=processed-data-topic

StreamBridge: Bridging the Gap

What if you want to send a message to Kafka from a REST Controller? A Supplier won't work because it's polled automatically. You need to send data on demand.

For this, Spring Cloud Stream provides StreamBridge.

@RestController
public class WebSourceController {

    @Autowired
    private StreamBridge streamBridge;

    @PostMapping("/data")
    public void sendData(@RequestBody String body) {
        // Sends the body to the "myDestination" binding (or topic directly)
        streamBridge.send("myDestination", body);
    }
}
