Lesson 1: The Functional Model

Since you are already familiar with messaging concepts (like topics and queues), we're going to dive straight into how Spring Cloud Stream abstracts them for modern Spring Boot applications.

The Core Idea

Spring Cloud Stream connects your application to a message broker (like Kafka or RabbitMQ) without you writing broker-specific code. It does this by binding standard Java functional interfaces to destinations (topics/queues).

In the modern paradigm, you don't use @StreamListener or custom interfaces. You simply declare beans of type java.util.function.

The Three Primitives

  • Supplier<O> (Produces data): Acts as a Source. It has no input, only output. Spring calls this continuously or on a trigger to send messages to a destination.
  • Consumer<I> (Consumes data): Acts as a Sink. It receives data from a destination but produces no output back to the broker.
  • Function<I, O> (Processes data): Acts as a Processor. It consumes data from one destination, transforms it, and produces output to another destination.

Code Example

Here is a complete message processor that receives a String, converts it to uppercase, and sends it out.

import java.util.function.Function;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class TextProcessor {

    @Bean
    public Function<String, String> uppercase() {
        return value -> {
            System.out.println("Received: " + value);
            return value.toUpperCase();
        };
    }
}

That's it. There is no Kafka or RabbitMQ code here. The infrastructure handles polling, serialization, and connection management.
