Lesson 7: Message Filters

We know how to transform, route, and enrich messages. But what if a message shouldn't be processed at all? What if it's invalid, expired, or doesn't meet a specific business requirement?

This is where the Message Filter pattern comes in.

The Role of a Filter

A Filter evaluates a message against a boolean condition. If the condition is true, the message is passed to the next channel. If the condition is false, the message is dropped (or optionally routed to a "discard" channel).

Filters in Java DSL

In the DSL, you use the .filter() method. Just like routers, you can filter based on the payload or the headers.

@Bean
public IntegrationFlow orderFlow() {
    return IntegrationFlow.from("newOrderChannel")
    
        // 1. Simple Filter: Only allow orders with a valid ID
        .<Order>filter(order -> order.getId() != null && !order.getId().isEmpty())
        
        // 2. Filter with a discard channel
        .<Order>filter(order -> order.getQuantity() > 0, 
            filterEndpointSpec -> filterEndpointSpec
                .discardChannel("invalidOrdersChannel")
        )
        
        // ... continue to routing ...
        .get();
}

Why not just use a Router?

You could write a Router that sends invalid messages to a null channel. However, conceptually, a Router is about "Where does this go?", while a Filter is about "Should this continue?". Using a Filter makes your intent much clearer to other developers reading your integration flow.
