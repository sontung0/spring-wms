Changing the Message

In integration flows, systems rarely speak the exact same language. System A might output a JSON string, but System B requires a Java Object. Or perhaps you receive an XML file and need to extract just one specific ID from it.

A Transformer is an endpoint responsible for converting a Message's content or structure and returning the modified Message. It is the translator of your integration flow.

Using Transformers in Java DSL

The Java DSL provides the .transform() method to apply transformations. You can use built-in transformers or write your own custom logic.

1. Custom Transformation (Lambdas)

The most common and flexible way to transform a payload is using a simple lambda expression. Spring Integration automatically handles extracting the payload, passing it to your function, and wrapping the result back into a new Message.

@Bean
public IntegrationFlow transformFlow() {
    return IntegrationFlow.from("inputChannel")
        // Transform a String payload to uppercase
        .transform((String payload) -> payload.toUpperCase())
        
        // Transform a String into a custom Object
        .transform((String payload) -> new Order(payload))
        
        .handle(System.out::println)
        .get();
}

2. Built-in Transformers

Spring Integration provides many built-in transformers for common tasks, especially when dealing with standard formats like JSON, XML, or Object Serialization. These are often accessed via factory classes like Transformers.

import org.springframework.integration.dsl.Transformers;

@Bean
public IntegrationFlow jsonFlow() {
    return IntegrationFlow.from("jsonInputChannel")
        // Convert JSON String to a Java Object (requires Jackson on classpath)
        .transform(Transformers.fromJson(Order.class))
        .handle(orderService, "processOrder")
        .get();
}

3. Transforming Headers

While .transform() is primarily for payloads, remember that we previously learned about .enrichHeaders(). Header enrichment is technically a form of transformation, specifically focused on the message metadata rather than the payload.
