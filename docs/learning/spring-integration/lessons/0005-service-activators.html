Lesson 5: Service Activators and POJOs

Up until now, our flow has only processed String objects, and we've written our logic right inside the configuration class using lambdas: .handle(m -> ...).

This is fine for simple examples, but in a real WMS application, you have Domain Objects (POJOs like Order, InventoryItem) and you want to delegate the actual work to existing Spring @Service beans.

The Service Activator

A Service Activator is the endpoint type that connects a Message Channel to a specific method on a Spring Bean. It takes the payload of the incoming message, passes it as an argument to your service method, and if your method returns something, it wraps that result in a new message and sends it to the next channel.

Connecting the DSL to your Services

Instead of passing a lambda to .handle(), we pass a reference to a Spring Bean and the name of the method we want to invoke.

// 1. You have a standard Spring Service
@Service
public class InventoryService {
    public void checkStock(Order order) {
        System.out.println("Checking stock for order: " + order.getId());
    }
}

// ...

// 2. In your IntegrationFlow
@Bean
public IntegrationFlow orderFlow(InventoryService inventoryService) {
    return IntegrationFlow.from("newOrderChannel")
        // The DSL will automatically extract the Order payload and pass it to checkStock
        .handle(inventoryService, "checkStock") 
        .get();
}
Magic Payload Extraction: You do not need to change your InventoryService to accept a Spring Integration Message<Order>. Spring is smart enough to extract the Order payload from the message and pass it to your method automatically. This keeps your business logic 100% decoupled from the messaging framework.
