Lesson 8: Error Handling

Mission: Master Spring Integration for spring-wms

The Challenge of Async Errors

In standard synchronous Java, if a method throws an exception, you catch it with a try/catch block. But in Spring Integration, messages often move asynchronously across threads or even different systems. If a Service Activator throws an exception on a background thread, your original calling code is long gone. How do you catch it?

Spring Integration solves this using the Error Channel pattern.

The ErrorMessage and errorChannel

When an exception occurs during message processing, Spring Integration catches it, wraps the original message and the exception into an ErrorMessage, and sends it to a designated error channel.

1. The Global errorChannel

By default, Spring Integration provides a global publish-subscribe channel named exactly errorChannel. If an exception is thrown in an asynchronous flow (like after a QueueChannel or a TaskExecutor), the framework automatically routes the error here.

2. Gateway Error Channels

When using a @MessagingGateway, you can specify an error channel specifically for that entry point. This allows you to handle errors differently depending on where the message originated.

@MessagingGateway(
    defaultRequestChannel = "inputChannel",
    errorChannel = "myCustomErrorChannel" // Route errors here
)
public interface MyGateway {
    void process(String payload);
}

3. Handling the Error

You handle errors just like any other message! You create an IntegrationFlow that listens to the error channel. The payload of the message will be a MessagingException, which contains both the cause (the actual exception) and the failedMessage.

@Bean
public IntegrationFlow errorHandlerFlow() {
    return IntegrationFlow.from("myCustomErrorChannel")
        .handle(message -> {
            MessagingException exception = (MessagingException) message.getPayload();
            System.err.println("Flow failed! Error: " + exception.getCause().getMessage());
            System.err.println("Failed message payload: " + exception.getFailedMessage().getPayload());
            // You could also send an alert, save to a dead-letter queue, etc.
        })
        .get();
}

Skill Check

Scenario: You have a flow that reads files from a directory and processes them asynchronously. If the processing fails (e.g., invalid file format), you want to move the file to a "failed" directory and send an email alert.

Question: How should you capture the exception to trigger this failure logic?

  1. Wrap the .handle() method in a standard Java try/catch block within the DSL definition.
  2. Configure an errorChannel on the flow's entry point and create a separate IntegrationFlow that reads from that channel to perform the failure logic.
  3. Use the .filter() method to check if the payload is an Exception.
